# 方法图谱与任务-方法对照 / Method Atlas and Task-to-Method Mapping

## Summary / 简介
This pack does not teach a single software workflow; it helps researchers understand how question types, evidence types, and methods map to one another.

这个知识包不教某一个软件怎么点，而是帮助用户理解不同问题、不同证据和不同方法之间的映射关系。

## When To Use / 何时调用
- Which methods are suitable for this type of question
- Give me a method atlas
- 给我一个方法图谱
- 什么问题适合什么方法

## Research Situations / 适用情境
- 不确定自己的问题更适合哪一类方法家族
- 看了很多方法名，却不知道它们各自擅长回答什么
- 想做方法比较，但总停留在工具名层面

## Core Concepts / 核心概念
- 方法图谱 / method atlas
- 任务-方法映射 / task-method routing
- 证据形态 / evidence form
- 设计风险 / design risk

## Frameworks / 方法框架
1. **Question-to-Method Matrix**: Compare method families by the kind of answer they can credibly support.
1. **Evidence-to-Analysis Matrix**: Map data forms to analysis strategies before choosing a named method.

## Expanded Workflow / 扩展工作流
1. 先定义你要交付的是模式、关系、机制还是过程。
2. 再判断可用证据属于数值、文本、案例、实验还是混合形态。
3. 据此筛出 2 到 3 个候选方法家族，而不是从几十种方法里盲选。
4. 比较每种方法的前提、风险和证据要求。
5. 把最终方法写回设计对齐表，而不是独立看待。

## Decision Rules / 判断规则
- 方法图谱用于路由，不用于代替设计思考。
- 先定证据形态，再比较方法家族。
- 如果你说不清方法家族的局限，通常也没真正选定它。

## Checklists / 检查清单
- What answer format do I need: pattern, relationship, mechanism, or process?
- What evidence form do I have or can realistically get?
- What method family is being excluded, and why?

## Common Pitfalls / 常见误区
- 把方法名当成答案
- 只看自己熟悉的方法
- 忽略方法的前提条件
- 把方法图谱误当成具体软件教程
